## 首先，AI 是什麼

廣義的 AI 範圍太大，這邊以近期我們最常用的 ChatGPT, Claude 這類的 AI 服務來舉例
這些 AI 模型們都是屬於 `大型語言模型 Large Language Model (LLM)`，以白話文來說就是按照使用者的輸入，來猜測接下來的應該要輸出什麼樣的內容
找到重點了嗎？ **猜測** 接下來的內容
說到底近代 AI 都只是在猜測使用者想要什麼樣的回答，更精準的說是在 `眾多選項中選出符合機率最高的選項`

> 當然現在的模型加入了許多最佳化策略，使得產出的結果並不會偏離太多

## 那又怎樣？

因此通常來說各個模型都會選比較保守的選項——因為越接近平均值，代表越不容易偏離重點
這也是朱宥勳老師跟謝宜安老師於 podcast 中所提到，他們在評學生文學獎會遇到的困境
按照他們近期的經驗，通常用 AI 寫的文章是介於五分上下的水準
聽起來不高也不低，不應該造成太多的問題才對？
但事實恰好相反，在學生文學獎中（特別是國高中的文學獎），通常只會有一兩篇超過這個分數、大約七分左右，剩下的就會直接斷崖式的落到三四分
也就是說這個用 AI 撰寫並參賽的文章，以分數來評大概會落在第三名到佳作的程度
這就讓他們很為難，因為以評審的角度來說，這篇文章就是比其他文章好，理應要給它一個獎項
不過從另一個角度來看，對自己認真寫文章的學生來說又不公平，如果他認真寫一篇文章最後輸給一個 AI，某方面來說他的努力是不是比不上那個敲幾個字就生出一篇文章的人？

> 在這集 podcast 中還有其他關於 AI 的問題，不過並不是本篇文章的重點因此先略過，歡迎有興趣的人自行前往聆聽

近期我自己也有類似的體悟，不過是關於 vibe coding 方面的

## 我遇到的問題

首先我並不是說 vibe coding 不好，也並不是要說非軟工相關就不應該寫 code

> Vibe Coding：泛指使用 AI 撰寫程式碼的行為，包括但不限於由使用者主筆並使用 AI 輔助，或是完全使用 AI 撰寫

這邊提一下我遇到的一些情境

### 情境ㄧ：純軟公司要死了

首先是我偶然看到的 influencer，他主要是講投資相關的，會在網路上發表他的投資心得
某一次他說到：「現在我不會考慮任何軟體公司的股票，例如 Adobe」
理由是因為他覺得現在 AI 非常強大，只要丟給一個有 idea 的人一個 AI 他就能做出來了

### 情境二：AI 會害我的帳密外洩

寫到這段的當天早上，加入了一個 Claude 相關的社團
本來是想說因為近期用 Claude Code 做了許多工作，加上還沒有摸透 Claude Code 的功能，加個社團稍微看一下別人是怎麼用 Claude Code 的好了
⋯⋯然後過五分鐘我就退出社團了，裡面有許多我看了不斷皺眉的貼文

像是有人提到他一用 Claude 帳密就外洩了，但從他的截圖來看根本就只是他被 Google 示警，而剛好 Google 列出外洩的密碼包含 Claude 這個網站的帳密
可以合理的推測是他在所有地方都用同一個密碼，而剛好被 Google 比對到的就是這一筆帳密紀錄而成了背鍋俠
但他又說他用 Claude 寫了超過十萬行的 code，絕對不是不會用電腦的人

### 情境三：我用 AI 寫出題目了！

這是從我學長那邊看到的情況，是發生在他與一個外系生之間的對話

那名外系生在不知道學長是資工生的情況下，告訴學長說他用 AI 寫出了一個大約等於大一程式設計期中考難度的 txt 文件處理，照學長所言那個方法可以被隨便產出來的測試資料擋掉
總之在他指揮 AI 瘋狂的抓 bug 之後完成了他的任務，並且不斷問學長他的看法如何，並且輕鬆的拒絕掉學長給他的建議
甚至還問學長說他會不會解費氏數列的 10000000000 項，但殊不知這在競程圈算是很入門的基礎題，就算隨便叫一個資工系的學生稍加引導也能寫出來

## 我的看法

對於第一個情境，我不會怪 influencer 不懂 AI 對於軟體產業的影響，甚至我認同他 `現在別投資純軟體公司` 這個結論
但對於他中間的論述我覺得完全是錯的，如果要做出一個 Photoshop 或是 Premiere 那麼容易，那 Adobe 應該分分鐘倒閉啊
現實就是：並不是隨便叫一個有 idea 的人來，丟給他 AI 他就能把那個 idea 做出來並且上架

最近我才看到一支短片，我猜情境應該是一個軟體工程師參加一個 AI coding 的活動，而台上的講者根本不是軟體產業的人
那個工程師想要對講者說，把產品做出來是一回事、將這個產品修到能上線並且處理上線後的問題又是另外一回事，同時他認為單純用 AI 產 code 但不 review 是不對的
然後台上的講者就開始跳針，她說：「所以我們這些不是軟體背景的就不應該做開發嗎？」
我的看法是不是不能做開發，而是你要對你開發的東西負責，你都看不懂的東西是要如何負責？

這又讓我想到之前發生了一個蠢事，有個做 AI 課程的人寫了某個服務，因為這個服務需要用到一點 AI 的功能於是她要求用戶在使用前要輸入用戶個人的 Gemini API key
先不提要求給 API key 是一件多詭異的操作，API key 代表有這個 key 的人都可以用 **你的** 帳號來操作 Gemini，而要知道 Gemini 是可以讀取你的 GMain 和 Google Drive 的，資料外洩的風險指數級上升（而且聽完後面的敘述應該會更覺得這個服務非常不安全
後續專案上線了，過了一個月她收到一萬鎂的 Gemini 帳單
結果才發現她的服務根本就沒有套用這些 API key，所有使用這項服務的人都是在使用她的帳號
隨後當事人於自己的臉書發文稱 Google 此舉是在詐騙她，因為這個服務完全是用 Gemini vibe coding 的，一定是 Google 的陰謀，讓 Gemini 寫一個會坑她錢的產品出來
後來被許多工程師洗臉，自己不會用工具還要怪廠商，更不用說現在所有的 AI 服務基本上都在使用者條款或是介面上標註 AI 的回覆可能出錯的 disclaimer
最後還在嘴硬，不肯認錯

> 想不到吧，這種人也可以出 AI 課程而且有人買單喔

至於第二個情境，我甚至懷疑他的十萬行是在灌水，去年我兩個兼職加起來寫的 code 都不到這個數字了，更不用說其中有一個網站的後端幾乎都是 AI 主筆，我只負責挑 bug 或是他處理不了的情況了
第三個情境我只能說是還不知道人外有人天外有天，用 AI 產了點東西就以為自己是軟體工程師了
搞不好第二跟第三的當事人在整個 vibe coding 的過程中只有提一些很發散的想法，然後就是不斷的 confirm permissions 就當作自己會寫程式了吧

在過去一年間才經歷從 `手寫 code + AI 輔助` 變成 `AI 主筆 + 我驗證 feature & CR` 的過程

## 我的經驗

起初我是拒絕訂閱任何 AI 服務的，因為說到底我還沒很信任 AI 寫的程式碼，除非是我火燒屁股了不然最多把它當成非常萬用的搜尋引擎，問完某個東西該怎麼使用後回頭繼續手搓程式碼
直到我積了一堆工作做不完，加上一直以來都很仰賴 auto-completion，如果有一個 AI powered 的 auto-completion 應該是挺不錯的
於是我就開始用 GitHub Copilot 了，反正第一個月免費試用，真的不滿意再取消，大不了被扣一個月的錢

> 當時的我還不知道 GitHub Copilot 有免費的學生方案，付了八個月的錢 = =

使用之後驚為天人，Copilot 甚至可以按照我寫的 definition 補出整個 function，而我只要補齊其中幾個寫錯的小地方就好
從這時候開始，我慢慢的使用越來越多 AI
而到夏天 Gemini 2.5 Pro 開放給免費用戶使用（雖然額度很少），我才開始用 AI 寫整個網頁，一開始是用 gemini-cli 寫這個網站的 [Coffee Timer](/coffee-timer/)
發現效果不錯後開始放手給 AI 做完整的工作，像是幾乎把整個服務後端丟給 AI 寫，我在功能完成後逐一驗證功能以及做 code review
